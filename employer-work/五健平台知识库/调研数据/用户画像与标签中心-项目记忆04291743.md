# 用户画像与标签中心

# 工程级开发记忆（可用于新对话恢复上下文）

版本：V1.0  
阶段：**系统原型评审阶段（Pre-Development Review Stage）**  
技术栈：

```text
Backend：Python（建议 FastAPI / Django）
Frontend：Vue
Database：MySQL / PostgreSQL
Architecture：平台级业务系统（标签中心 / 用户画像中心）
```

---

# 一、系统定位（必须统一）

```text
系统名称：

用户画像与标签中心
```

系统角色：

```text
平台级能力中心（Platform Capability Center）
```

不是：

```text
业务系统
```

而是：

```text
为多个业务系统提供标签与人群能力
```

例如：

```text
体检系统
随访系统
学校系统
健康管理系统
公共卫生系统
```

---

# 二、业务核心概念（已统一）

重要：  
系统中不再使用：

```text
Entity（主体）
```

统一使用业务名称：

```text
服务对象（Service Object）
```

这是当前项目的正式业务术语。

---

# 三、系统模块结构（已冻结）

当前系统采用 **8 个核心模块**：

```text
1 服务对象统一管理（Service Object Registry）

2 服务对象关系管理（Service Object Relation）

3 标签体系管理（Tag Dictionary）

4 标签规则引擎（Rule Engine）

5 标签计算与分配（Tag Assignment）

6 用户画像管理（Profile Management）

7 人群管理（Audience Management）

8 标签应用中心（Tag Service）
```

这是：

```text
系统一级模块结构（不可随意改变）
```

# 五、最重要的数据建模决策（必须记住）

---

## 1）id 与 code 的区别（团队必须统一）

```text
id

系统内部唯一
数据库主键
不可修改
不可展示
用于关联

entity_code（服务对象编码）

业务唯一
可展示
可搜索
可导出
可外部使用
```

---

## 示例

```text
id = 10000001
entity_code = SO202400001
```

---

## 2）服务对象必须支持多个标识（核心设计）

服务对象不能只有一个 ID。

必须支持：

```text
一个服务对象
多个标识
```

---

## 标识表示例

```text
entity_identifier
```

字段：

```text
id
entity_id
identifier_type
identifier_value
is_primary
```

---

## 标识类型（已确定）

```text
ID_CARD
身份证

STUDENT_NO
学号

PHONE
手机号

OPENID
微信

MEDICAL_NO
就诊号

SOCIAL_NO
社保号
```

# 六、扩展字段设计（已确定）

系统必须支持：

```text
动态字段（Dynamic Attribute）
```

表：

```text

```

---

## 示例字段

```text

class
```

因为：

```text

```


