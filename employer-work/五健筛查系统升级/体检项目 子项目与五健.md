我们把这三个问题按**业务本质 → 数据模型抽象 → 表设计落点**的顺序来严谨梳理，这样后续你设计 `screening_rules` 表时会非常稳，不会反复推翻。

---

# 问题1：五健本质是体检？

**结论：是，但不是普通体检，而是“专项筛查型体检体系”。**

可以用一句工程化定义：

> **五健 = 面向儿童青少年重点健康问题的标准化筛查型体检（Screening-oriented Health Examination）**

它在系统设计上属于：

```text
体检系统（Health Examination System）
    └── 专项筛查体系（Screening Program）
            └── 五健（体重 / 视力 / 心理 / 脊柱 / 口腔）
```

---

## 从业务属性上看，五健具有 4 个典型特征

|属性|普通体检|五健筛查|
|---|---|---|
|目标|健康评估|风险发现|
|粒度|项目级|指标级|
|频率|不固定|周期性（每年）|
|结果|诊断/建议|风险等级/随访|

---

## 在系统建模中的定位（非常关键）

五健在数据模型中不是一个单独系统，而是：

```text
体检系统
    └── Screening（筛查）
            └── Screening Program（五健）
                    └── Screening Item（视力）
                            └── Screening Indicator（左眼视力）
```

也就是说：

> 五健 = 体检系统中的一种 **筛查型业务模式**

这点会直接决定 `screening_rules` 的设计粒度。

---

# 问题2：体检系统中是不是分为组合项、子项？

**结论：是，而且这是标准行业结构。**

你描述的是完全正确的，只需要把术语规范一下。

---

# 标准三级结构（强烈建议固定）

```text
Screening Program（筛查方案）

    Screening Item（组合项 / 项目）
        例如：
            内科
            视力
            口腔
            心理
            脊柱

        Screening Indicator（子项 / 指标）
            例如：

            内科：
                心率
                心音
                血压

            视力：
                左眼视力
                右眼视力
                沙眼
                散光

            脊柱：
                前屈试验
                脊柱侧弯角度
```

---

## 用数据库实体表达（推荐标准）

```text
screening_program
    id
    name

screening_item
    id
    program_id
    name
    type

screening_indicator
    id
    item_id
    name
    data_type
    unit
```

---

## 非常关键的一句话

> **规则判断一定发生在“指标（Indicator）层”，而不是“组合项（Item）层”。**

因为：

```text
异常 是由指标值触发的
不是由项目名称触发的
```

例如：

```text
左眼视力 < 4.8
BMI ≥ 28
PHQ-9 ≥ 10
脊柱侧弯角度 ≥ 10°
```

这些全部是：

> Indicator 级规则

# 问题3：screening_rules 放在组合项目上？还是子项目上？

**结论（非常明确）：**

> screening_rules 必须绑定在 子项目（Indicator）上。

不是 Item。  
不是 Program。  
就是 Indicator。

---

# 为什么必须放在子项目上？

我们用一个真实例子。

---

## 例子：视力筛查

```text
组合项（Item）：
    视力

子项（Indicator）：
    左眼视力
    右眼视力
    散光
    沙眼
```

---

如果规则放在 **Item（视力）**

你只能写：

```text
视力异常
```

这是：

> 不可执行规则  
> 不可计算规则  
> 不可解释规则

---

而真正的规则是：

```text
左眼视力 < 4.8
右眼视力 < 4.8
散光 = 是
沙眼 = 阳性
```

这些都是：

> Indicator-level rule

---

# 正确的数据关系（标准）

```text
screening_indicator
        │
        │ 1
        │
        │ N
screening_rules
```

---

# 推荐的 screening_rules 表结构（标准版）

```sql
screening_rules
----------------------------
id

indicator_id
rule_name

operator
threshold_value

unit

risk_level

gender
min_age
max_age

is_active

version

created_at
updated_at
```

---

# 字段解释（关键）

|字段|含义|
|---|---|
|indicator_id|绑定哪个子项（核心）|
|operator|< > <= >= = between|
|threshold_value|阈值|
|risk_level|正常 / 预警 / 高风险|
|gender|男 / 女 / ALL|
|age_range|年龄段|
|version|规则版本|

---

# 真实规则示例

```text
规则：
男生 7–12 岁
BMI ≥ 24
= 超重
```

在表里是：

```text
indicator_id = BMI

operator = >=

threshold_value = 24

gender = male

min_age = 7
max_age = 12

risk_level = overweight
```

# 一个重要的补充：Item 级规则有没有？

有，但不是 screening_rules。

而是：

```text
risk_evaluation_rules
```

或者：

```text
item_aggregation_rules
```

---

## 例如：视力整体异常判断

```text
如果：

左眼视力异常
或
右眼视力异常

则：

视力异常
```

这是：

> 聚合规则（Aggregation Rule）

不是：

> 指标阈值规则（Indicator Rule）

---

# 所以最终是三层规则体系（非常标准）

```text
Level 1
Indicator Rule
（screening_rules）

左眼视力 < 4.8

──────────────

Level 2
Item Aggregation Rule

左眼异常 OR 右眼异常
= 视力异常

──────────────

Level 3
Disease Mapping Rule

视力异常
→ 近视风险
```

这和你现在的：

> Tag–Disease 体系

是完全兼容的。

# 给你的最终设计结论（可直接用于评审）

```text
1）五健本质
五健属于体检系统中的专项筛查体系（Screening Program）

2）体检结构
标准为三级：

Program（筛查方案）
    Item（组合项）
        Indicator（子项）

3）screening_rules 的绑定对象
必须绑定在：

screening_indicator（子项）

不是：

screening_item（组合项）
```
