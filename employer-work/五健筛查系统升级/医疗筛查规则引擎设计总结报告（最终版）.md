# 医疗筛查规则引擎设计总结报告（最终版）

# 一、项目背景

本次对话围绕：

# 医疗筛查系统规则引擎

进行系统化设计与架构收敛。

目标系统主要面向：

- 五健筛查
    
- 学生健康体检
    
- 儿童青少年健康管理
    
- AI健康报告
    
- 健康画像平台
    
- 阳性结果判定
    
- 标签与风险管理
    

核心目标：

```text
体检原始数据
→ 异常指标
→ 疾病规则判断
→ 阳性结果
→ 标签
→ 画像
```

---

# 二、规则系统总体定位

最终确定：

# 疾病规则系统

是：

# 独立规则中心

而不是：

# 挂靠在体检项目中的简单配置。

---

# 三、整体架构模型（最终版）

最终形成如下结构：

```text
体检项目
    ↓
体检子项结果
    ↓
异常指标
    ↓
疾病规则引擎
    ↓
疾病结果
    ↓
疾病知识库
    ↓
标签系统
    ↓
画像系统
```

---

# 四、规则引擎核心结构（最终确认）

最终规则结构：

```text
疾病
  ↓
规则路径（Rule Path）
  ↓
适用范围（Scope）
  ↓
阳性判断条件（Conditions）
```

---

# 五、规则路径（Rule Path）

规则路径：

# 本质是一条完整疾病判定规则。

规则路径之间：

# OR关系

表示：

```text
满足任意路径即可命中疾病
```

---

# 六、适用范围（Scope）

Scope：

# 描述规则适用于谁

而不是疾病判断逻辑本身。

典型字段：

|字段|示例|
|---|---|
|年龄|7~18岁|
|性别|男|
|学段|小学|
|地区|城区|
|年级|一年级|

---

# 七、Scope核心原则

---

# 1. Scope属于规则路径级

即：

```text
一个规则路径
对应一套Scope
```

而不是：

```text
一个条件对应一个Scope
```

---

# 2. Scope天然AND

例如：

```text
年龄满足
AND
性别满足
AND
学段满足
```

必须全部成立。

---

# 3. 不同规则路径允许不同Scope

例如：

---

## 幼儿规则

```text
年龄<72月
```

---

## 学龄规则

```text
7岁≤年龄<18岁
```

---

这是正确且必要的。

---

# 八、阳性判断条件（Conditions）

Conditions：

# 真正参与疾病判定。

例如：

|条件|
|---|
|BMI ≥ P85|
|裸眼视力 < 5.0|
|SPH ≤ -0.50|
|CYL ≤ -0.75|

---

# 九、条件结构（最终确认）

所有条件统一采用：

```text
字段 + 运算符 + 值
```

标准结构。

例如：

```text
BMI >= P85
身高 < 160
SPH <= -0.50
```

---

# 十、原子条件（Atomic Condition）

最终确定：

# 所有复杂逻辑必须拆成原子条件。

例如：

---

# 区间表达：

```text
100 <= 身高 < 160
```

实际拆分为：

```text
身高 >= 100
AND
身高 < 160
```

---

# 十一、逻辑关系设计（最终确认）

---

# 1. 条件之间：

默认：

# AND

---

# 2. 规则路径之间：

默认：

# OR

---

# 3. 当前阶段：

不支持：

```text
自由混合AND/OR表达式
```

原因：

- 医疗人员无法维护
    
- UI复杂度爆炸
    
- 容易产生歧义
    

---

# 十二、值类型（Value Type）

本次对话最重要升级之一：

# 增加“值类型”

最终结构：

|字段|示例|
|---|---|
|field|BMI|
|operator|>=|
|value_type|percentile|
|value|P85|

---

# 十三、值类型分类（最终建议）

---

# 1. FIXED_VALUE（固定值）

例如：

```text
BMI > 28
```

---

# 2. PERCENTILE_STANDARD（百分位标准）

例如：

```text
BMI >= P85
```

---

# 3. REFERENCE_RANGE（参考范围）

例如：

```text
视力 < 正常下限
```

---

# 4. AGE_SEGMENT_STANDARD（年龄分段标准）

例如：

```text
不同年龄视力标准
```

---

# 5. GROWTH_CURVE_STANDARD（生长曲线标准）

例如：

```text
BMI Z-Score
```

---

# 十四、P85不是固定值（核心结论）

本次对话重点结论：

# P85不是具体数字

而是：

# 动态参考标准代码。

例如：

---

# 7岁男生：

```text
P85 = 21.3
```

---

# 10岁男生：

```text
P85 = 23.7
```

---

因此：

```text
BMI >= P85
```

实际执行过程：

---

# Step1

获取：

```text
年龄
性别
```

---

# Step2

查询参考标准表：

```text
BMI_P85
```

---

# Step3

获得真实阈值：

```text
21.3
```

---

# Step4

执行比较：

```text
BMI >= 21.3
```

---

# 十五、参考标准引擎（Reference Standard Engine）

最终确认：

# 医疗规则系统

必须独立：

# 参考标准引擎

负责：

- P85/P95
    
- 年龄标准
    
- 参考范围
    
- 生长曲线
    

---

# 十六、规则系统与参考标准系统分离

最终架构：

```text
规则引擎
    ↓
引用参考标准
    ↓
参考标准引擎
    ↓
返回真实阈值
    ↓
执行规则
```

---

# 十七、规则优先级（最终结论）

最终结论：

# 当前阶段：

# 不需要条件优先级

原因：

当前模型已经固定：

```text
组内AND
组间OR
```

不存在：

```text
AND/OR混合优先级问题
```

---

# 十八、未来可能需要的优先级

未来：

可能需要：

# 疾病结果优先级

例如：

|疾病|priority|
|---|---|
|重度肥胖|100|
|肥胖|90|
|超重|80|

用于：

```text
多疾病同时命中
```

时的结果控制。

# 二十、规则复杂度控制原则（核心）

本次对话最终形成的重要思想：

# 医疗规则系统：

# 必须限制复杂度

因此：

---

# 不做：

- 无限嵌套表达式
    
- AST编辑器
    
- 通用Drools规则平台
    
- 自由逻辑表达式
    

---

# 优先：

- 有限医疗DSL
    
- 固定规则结构
    
- 医疗人员可维护
    
- 可解释性
